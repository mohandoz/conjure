  <action>Insert a new "Check version consistency" step into the test job in .github/workflows/ci.yml. The step must be inserted AFTER the existing "Validate JSON" step and BEFORE the "Run kit test suite" step.

The new step reads the VERSION file and compares it against both manifest JSON files. It fails CI (exit non-zero) if any version field drifts.

Critical path detail from RESEARCH.md: the version field that matters in marketplace.json is `.plugins[0].version` (NOT `.version` at root level). The plugin.json version is `.version` at root.

Step to insert:
```yaml
      - name: Check version consistency
        run: |
          ver=$(cat VERSION)
          mkt_ver=$(jq -r '.plugins[0].version // empty' .claude-plugin/marketplace.json)
          plg_ver=$(jq -r '.version // empty' .claude-plugin/plugin.json)
          rc=0
          [ "$mkt_ver" = "$ver" ] || { echo "FAIL: marketplace.json .plugins[0].version ($mkt_ver) != VERSION ($ver)"; rc=1; }
          [ "$plg_ver"  = "$ver" ] || { echo "FAIL: plugin.json .version ($plg_ver) != VERSION ($ver)"; rc=1; }
          [ "$rc" -eq 0 ] && echo "OK: all version fields match $ver"
          exit "$rc"
```

The `// empty` jq fallback ensures that if the field is missing, the variable is empty (not "null"), making the string comparison fail cleanly with a useful error message rather than comparing "null" against the version string.

Do not modify any existing steps. Do not change the "Validate JSON" step — it already covers .claude-plugin/*.json for JSON syntax. This new step is for semantic version-consistency only.</action>

  <action>Insert two more steps into the test job in .github/workflows/ci.yml, immediately after the "Check version consistency" step added in Task 1. Both steps must appear BEFORE "Run kit test suite".

STEP 1 — Install claude CLI (per D-02: install failure = CI failure). Use the signed apt repository method from RESEARCH.md (preferred over curl|bash in CI because package manager signatures are verified automatically):

```yaml
      - name: Install claude CLI
        run: |
          sudo install -d -m 0755 /etc/apt/keyrings
          sudo curl -fsSL https://downloads.claude.ai/keys/claude-code.asc \
            -o /etc/apt/keyrings/claude-code.asc
          echo "deb [signed-by=/etc/apt/keyrings/claude-code.asc] https://downloads.claude.ai/claude-code/apt/stable stable main" \
            | sudo tee /etc/apt/sources.list.d/claude-code.list
          sudo apt-get update -qq
          sudo apt-get install -y claude-code
          claude --version
```

The final `claude --version` serves as an install verification gate — if the binary is not on PATH the step fails immediately (honoring D-02: install failure = CI failure). Do NOT add `continue-on-error: true` or any error suppression.

STEP 2 — Validate plugin manifests (per D-01 and resolved D-03 from RESEARCH.md). Two separate invocations are required because `claude plugin validate .` only validates marketplace.json when both files are present in .claude-plugin/:

```yaml
      - name: Validate plugin manifests
        run: |
          claude plugin validate .
          claude plugin validate .claude-plugin/plugin.json
```

Do NOT use `--strict` flag (per RESEARCH.md Pitfall 2 — strict promotes warnings to errors and should only be used when all warnings are resolved). The validate step fails on schema errors (exit 1) which is the CI failure condition per MKTPL-03.
