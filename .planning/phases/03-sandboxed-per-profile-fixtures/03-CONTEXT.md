# Phase 3: Sandboxed Per-Profile Fixtures - Context

**Gathered:** 2026-05-24
**Status:** Ready for planning

<domain>
## Phase Boundary

Create one committed, audited fixture per stack profile under `tests/fixtures/<profile>/`.
Each fixture contains the Claude Code harness layer (`.claude/`) plus a single
profile-appropriate manifest stub (package.json, pom.xml, Cargo.toml, etc.).

Also create `tests/fixtures/_broken/` — a fixture that deliberately exceeds the
CLAUDE.md 100-line cap, proving the audit suite can catch regressions.

Add `tests/lib/sandbox.sh` — a sourced bash helper that copies any fixture to a
temp dir and exports `HOME`/`XDG_CONFIG_HOME`/`CLAUDE_CONFIG_DIR`/`PATH` to that
copy, isolating fixture audit runs from the developer's real `$HOME`.

Add `scripts/regen-fixtures.sh` — runs `conjure init --profile <p>` against each
profile manifest seed and overwrites `tests/fixtures/` with the regenerated output.

Requirements: TEST-01, TEST-02, TEST-04.

</domain>

<decisions>
## Implementation Decisions

### Fixture Content
- **D-01:** Each fixture contains the `.claude/` harness layer (CLAUDE.md, skills/,
  agents/, hooks/, settings.json) **plus one manifest stub** appropriate to the
  stack profile. No src/ tree — manifest-only is sufficient to trigger
  profile-specific preflight checks.
- **D-02:** Manifest stubs per profile:
  - `ts-next` → `package.json`
  - `java-spring` → `pom.xml`
  - `rust-axum` → `Cargo.toml`
  - `go-gin` → `go.mod`
  - `python-fastapi` → `requirements.txt`
  - `node-nest` → `package.json`
  - `monorepo` → `package.json` (root workspace manifest)
  - `polyglot` → `package.json` (representative; polyglot has no single manifest type)
  - `data-science` → `requirements.txt`
- **D-03:** Hook wiring in fixtures uses `node .mjs` relative paths (Phase 1
  D-01/D-02) — never `bash .sh`.

### Sandbox Isolation
- **D-04:** Isolation helper lives at `tests/lib/sandbox.sh` — a sourced bash
  library. Phase 4's test runner sources it alongside `tests/lib/` for other
  shared logic. Follows the `lib/` pattern from Phase 2 (`lib/mutate.sh`).
- **D-05:** `sandbox.sh` copies the fixture directory to a `mktemp -d` temp dir,
  then exports: `HOME`, `XDG_CONFIG_HOME`, `CLAUDE_CONFIG_DIR`, and a restricted
  `PATH` (CONJURE_HOME/cli + system) all pointing to the temp copy. The original
  committed fixture is never mutated.
- **D-06:** Cleanup: `trap 'rm -rf "$SANDBOX_DIR"' EXIT` inside sandbox.sh to
  ensure temp dirs are removed even on failure.

### Broken Fixture
- **D-07:** Broken fixture lives at `tests/fixtures/_broken/` (underscore prefix
  sorts first, clearly signals non-green intent).
- **D-08:** Deliberately violates the **CLAUDE.md 100-line size cap** — pad
  `CLAUDE.md` to 105+ lines. Deterministic, easiest to assert against exact
  finding string from audit output.
- **D-09:** Expected findings encoded in `tests/fixtures/_broken/EXPECT` — one
  grep pattern per line. Phase 4's golden-file runner asserts each pattern
  matches audit output. Same EXPECT format used for all fixtures (green fixtures
  can have an EXPECT file too, asserting `0 errors` or similar).

### Fixture Generation
- **D-10:** Fixtures are **generated by `conjure init --profile <p>`** against a
  minimal seed (just the manifest stub) and committed. Ensures fixtures always
  reflect actual `init` output rather than hand-crafted approximations.
- **D-11:** `scripts/regen-fixtures.sh` is the documented regen command. Run
  manually when profiles change. The script: for each profile, creates a temp
  seed dir with just the manifest stub, runs `conjure init --profile <p>` against
  it, and copies the output over `tests/fixtures/<p>/`.
- **D-12:** A `# GENERATED — do not edit directly; run scripts/regen-fixtures.sh`
  comment in each fixture's CLAUDE.md header signals to contributors not to edit
  fixtures by hand.

### Claude's Discretion
- Exact content of each manifest stub (e.g., minimal package.json has just
  `{"name":"fixture","version":"0.0.0"}` or something slightly richer — whatever
  triggers the profile's preflight.sh reliably).
- Whether `scripts/regen-fixtures.sh` accepts a `--profile <p>` flag to regen
  a single profile (useful during development) in addition to the all-at-once
  invocation.
- The exact grep pattern in `tests/fixtures/_broken/EXPECT` (e.g.,
  `CLAUDE.md.*exceeds.*100` or `size.*cap` — match what `conjure audit` actually
  emits).

</decisions>

## Existing Code Insights

### Reusable Assets
- `tests/run.sh:194` — existing `mktemp -d` + env-override pattern for dry-run
  test; `sandbox.sh` should follow the same idiom and be extractable from this.
- `lib/mutate.sh` — establishes the `lib/` sourced-library pattern; `tests/lib/`
  follows the same convention at the test level.
- `profiles/ts-next/apply.sh` — references `source "$CONJURE_HOME/lib/mutate.sh"`;
  shows how lib/ is resolved from `$CONJURE_HOME`.

### Established Patterns
- `mktemp -d` + `trap 'rm -rf "$DIR"' EXIT` — already used in tests/run.sh for
  temp target dirs; mirror in sandbox.sh.
- `CONJURE_HOME` env var already set by CLI before calling child scripts —
  sandbox.sh must preserve it when setting up the isolated env.
- `node .mjs` relative paths in settings.json hook wiring (Phase 1) — all 9
  fixtures must use this, never `bash .sh`.
- Phase 2 dry-run output: `[dry-run] N mutations skipped` — Phase 4 will grep
  for these in fixture runs; fixtures themselves don't emit them but sandbox.sh
  must not interfere with the output stream.

### Integration Points
- `tests/lib/sandbox.sh` sourced by `tests/run.sh` (new fixture audit section)
  and eventually by Phase 4's golden-file loop.
- `scripts/regen-fixtures.sh` invokes `cli/conjure init --profile <p>` per
  profile; must be idempotent (safe to run multiple times).
- `tests/fixtures/_broken/EXPECT` consumed by Phase 4's EXPECT comparison loop —
  establish the format now so Phase 4 can generalize it to all fixtures.

</code_context>

<specifics>
## Specific Ideas

- `sandbox.sh` interface: `sandbox_run <fixture_dir> <command...>` — copies fixture
  to temp, sets env, runs the command, captures output, tears down. Or a simpler
  `sandbox_setup <fixture_dir>` that sets env vars and leaves teardown to the
  caller's trap. Prefer the setup-only form so callers control capture.
- `_broken/EXPECT` can start with just one pattern (the size-cap violation line)
  and be extended by Phase 4 for more findings. Keep it minimal for Phase 3.
- `scripts/regen-fixtures.sh` should print `[regen] <profile>` per fixture so
  it's greppable and consistent with `[dry-run]` prefix established in Phase 2.

</specifics>
